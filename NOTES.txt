Status overview (TL;DR)
-----------------------

This is the rolled-up state. See the Confidence summary at the bottom
for the full per-signal log with cross-validation.

Pack & topology (CONFIRMED via [vendor GUI](https://2758547507-files.gitbook.io/~/files/v0/b/gitbook-x-prod.appspot.com/o/spaces%2FKQJ6j30UbUXhtNJMkJQP%2Fuploads%2Fffjy1KG6NFQEvQqlw8zn%2FBMS%20Update%20Error%20and%20Data%20Extraction%20-%20MS%20Soundon%20Battery.pdf?alt=media&token=eb0bc6a8-46a8-40db-b428-833f94d925dd)):
    20 series cells, 7 temperature probes, NMC chemistry, 300 Ah / 72 V
    nominal.

Decoded (CONFIRMED):
    * F113..F13C  per-cell voltages, BE u16 mV.
    * F155..F15E  per-module temps, u8 with +40 °C offset.
    * F102F3      cell min/max mV summary, 1-based cell-number indexing,
                  spread.
    * F100F3      pack voltage (V = b * 0.1 + 76.8) and signed pack
                  current (16-bit BE, 0x7D00 bias, 0.1 A/bit).
    * F100D0      VC heartbeat (0x00 init / 0x0C ready).
    * F108F3      dashboard BMS warning code bitmap in the last byte
                  (codes 124, 140, 142..146).
    * FF50E5      charger telemetry — status, output V, output A.
                  V/A only meaningful while status = 0x03.
    * FF21CA      motor telemetry — RPM (magnitude, with 0x0C80 bias),
                  throttle pedal, controller temp, packed transmission
                  state (byte 7: high nibble = range gear 1..3, low
                  nibble = F/N/R lever as 0/4/8). Suppressed while
                  charging.

Partially decoded (TENTATIVE):
    * F106    running-mode vocabulary (0x00 init / 0x80 standby /
              0x45 ready) — partial.
    * F107    limits-frame layout (current and voltage limits with
              0x2710 sentinel).
    * F108    additional fault info beyond the byte-7 code bitmap;
              bit-to-code mapping for the other bytes open.
    * F100F3  byte 5 (data[4]) = BMS-published SOC, calibrated only on
              the top 10% (raw 224→90%, raw 250→100%) from charging-
              120V-90ish-to-100.asc. Plus residual status / limit
              candidate bytes near the end of the payload.
    * FF21CA  three-state byte tied to startup calibration.

Open (UNKNOWN):
    * SOH / alarm-word host PGN.
    * KL15 / wake-up status bit location.
    * F104 byte-level decode (temp min/max summary).
    * Whether a separate drivetrain bus carries inverter / PTO telemetry.

Solved 2026-05-13:
    * Motor-controller error-code channel — codes ride standard
      J1939 DM1 (FECA from SA 0xCA); the SPN value equals the
      dashboard-displayed MC code (12..99 per the operator manual).
      The cluster prepends "MC" based on the source address. DM1
      was empty in every prior capture simply because no MC faults
      had occurred organically — not because the codes lived
      elsewhere. Confirmed via live CAN injection (util/mc_inject.py)
      with SPN 12, 47, and 99 → dashboard showed MC12, MC47, MC99.
      A populated DM1 from SA 0xF3 (BMS) was ignored by the
      cluster, confirming the cluster has subsystem-specific
      decoders rather than a unified DM1 path: MC uses DM1, BMS
      uses the proprietary F108 bitmap. Latch quirk: the cluster
      holds DTCs until key cycle; an absence of DM1 does not
      unlatch them as standard J1939 would prescribe.


Tractor / pack context
----------------------

* Tractor: Solectrac electric tractor (~70 V class).
* Pack chemistry: NMC (vendor screenshot labels it "NiCoMn"), confirming
  what the OCV-curve shape already suggested.
* Pack topology: **20 series cells** (s = 20), now CONFIRMED both from
  per-cell PGN coverage and from the vendor BMS GUI ("Cells in series: 20",
  "Online cell count: 20"). Module-temperature frames reserve up to 80
  channels across F155..F15E but the pack has **7 temperature probes**
  ("Temp. probe count: 7" / "Online temp. count: 7" in the GUI) — anything
  beyond the first 7 channels in `temps.csv` is padding/noise.
* Pack capacity / ratings (from the BMS GUI, not derivable from CAN alone):
    - Rated capacity: 300 Ah
    - Rated total voltage: 72.0 V nominal
    - Rated discharge current: 300 A
    - Rated DC charge: 78 A
    - Rated AC charge: 39 A
* SOC at time of capture: ~40% (operator-reported) for the original
  recorded-data/*.csv set, ~100% for the asc/*.asc set.
* Ambient: ~55 °F (~13 °C); tractor was outdoors.
* Drive-train state: most asc/*.asc captures were taken with the tractor
  in **neutral**. "Full throttle" in those filenames refers to throttle
  pedal position, not actual mechanical load on the wheels. The
  loader-arms.asc / loader2.asc captures (2026-05-09) added hydraulic
  load with the motor controller silent — together with driving.asc
  these were enough to nail down the F100F3 pack-current encoding
  (see the F100F3 section).
* The motor controller / drive ECU **is** on this bus after all (SA 0xCA),
  it just doesn't speak standard J1939 EEC1. Its telemetry is carried in
  the proprietary B frame FF21CA — see the FF21CA decode section below.
  This was confirmed empirically: accellerate-decelerate.asc produces a
  clean 0 → ~2500 → 0 RPM trace from FF21CA bytes 3..4 LE that matches
  the operator's reported reading of "near 2500" at full throttle.

  Whether there is *also* a separate drivetrain bus carrying the
  inverter / PTO is still open. The dashboard could be reading RPM
  directly from FF21CA on this bus, or it could be a dual-bus gateway
  with FF21CA being a re-broadcast. Either way, the most operator-
  visible drive-side data is reachable from here.

PGN decode notes (BMS, SA = 0xF3)
---------------------------------

All scalings below were derived empirically. Where a scaling is consistent
with sane physical units across multiple captures it's marked CONFIRMED;
otherwise it's TENTATIVE.

F113..F13C — "BMS cell voltages N..N+3"
    8 bytes = 4 × big-endian uint16 in millivolts.
    F113 = cells 0..3, F114 = cells 4..7, ... F13C = cells 164..167.
    Only the first 20 cells of F113..F117 have nonzero / sensible
    values for this pack; the rest are 0xFFFF (sentinel) or 0.
    CONFIRMED: BE u16 mV. Cells read ~3.6–3.7 V at ~40% SOC.

F155..F15E — "BMS module temps 8N..8N+7"
    8 bytes = 8 × uint8 with J1939 +40 °C offset (i.e. raw 53 = 13 °C).
    F155 = channels 0..7, F156 = 8..15, ... F15E = 72..79.
    Most channels are 0xFF (not present); a small number reflect ambient
    and were what let us pin down the +40 °C convention.
    CONFIRMED: u8, offset 40 °C.

F102F3 — "BMS cell min/max summary"
    Carries pack-wide cell statistics. We use:
      * bytes 1..2 (BE u16) = max-cell mV
      * bytes 3..4 (BE u16) = min-cell mV
      * byte 5            = max-cell **number, 1-based** (CONFIRMED)
      * byte 6            = min-cell **number, 1-based** (CONFIRMED)
      * byte 8            = status/flag bits
      * spread = max - min (mV) — derived
      * a coarse pack-voltage estimate (sum of 20 × cell average), kept
        as a sanity check
    CONFIRMED for max/min/spread.

    Indexing convention (resolved by row-order correlation between F102
    frames and surrounding F113..F117 cell-voltage frames in
    recorded-data/charging.csv):

      * byte5/byte6 are **1-based** cell numbers as the BMS GUI displays
        them (e.g. "Max cell #19").
      * To map to the parser's 0-based `cell_index` in cells.csv,
        subtract 1: BMS cell N = decoder cell_index N-1.
      * Cross-check: in the smallest-spread frames (3 mV) of
        recorded-data/charging.csv, F102 reports byte5=16, byte6=1,
        max=3649 mV, min=3646 mV. The contemporaneous per-cell snapshot
        shows decoder cell_index 15 at 3649 mV (= BMS cell #16 ✓) and
        decoder cell_index 0 as the lowest at 3645 mV (= BMS cell #1 ✓).
      * The vendor screenshot's "Max cell #19 / Min cell #2" therefore
        corresponds to decoder cell_index 18 and cell_index 1.

    Minor caveat: F102's reported min_mv is occasionally 1 mV higher
    than the actual lowest voltage seen in the per-cell PGN snapshot
    around it (e.g. 3646 vs 3645). Two plausible explanations: brief
    timing skew between the F102 summary computation and the per-cell
    sweep, or light filtering inside the BMS. The *index* still
    correctly points at the right cell, just with a slightly stale mV.

    Tie-breaking observation: when several cells are tied at the max,
    the BMS appears to report the lowest-index winner. Across 124
    smallest-spread frames where decoder cell_index {15, 17, 18} were
    all tied at 3649 mV, F102 byte5 was always 16 (= cell #16, i.e.
    decoder index 15, the lowest of the tied set).

    Smallest spread observed across all captures: **3 mV** (124 frames,
    all in recorded-data/charging.csv).

F100F3 — "BMS pack status" (CONFIRMED pack-current source)
    Across all asc/*.asc captures (5 in-neutral + driving.asc with real
    load, plus loader-arms.asc / loader2.asc which exercise hydraulic
    load with the motor controller silent) the byte structure is:

      byte 1 (data[0]): 0x03                  — constant
      byte 2 (data[1]): pack terminal voltage — CONFIRMED.
                                                 V = b * 0.1 + 76.8.
                                                 Range observed across
                                                 all captures: 0x35..0x42
                                                 → 82.1..83.4 V. Drops
                                                 INVERSELY with load
                                                 (sag under driving /
                                                 hydraulic / inverter
                                                 contactor draw).
                                                 Anchored by linear
                                                 regression vs
                                                 20*mean(cell mV)
                                                 across 24 captures
                                                 (residuals < 0.55 V)
                                                 and cross-checked
                                                 against the FF50
                                                 charger frame which
                                                 uses an identical
                                                 encoding (R^2 = 0.986
                                                 across 2863 active-
                                                 charging frames in
                                                 charging-120V-90ish-
                                                 to-100.asc).
      bytes 3-4 (data[2..3]):                 — SIGNED PACK CURRENT,
                                                 16-bit big-endian, 0.1
                                                 A/bit, biased so that
                                                 raw 0x7D00 = 0 A.
                                                 CONFIRMED — see below.
      byte 5 (data[4]): 0xF8..0xFA (248..250) — BMS-published SOC.
                                                 TENTATIVE. Calibrated
                                                 on charging-120V-90ish-
                                                 to-100.asc: raw 224 at
                                                 90% and raw 250 at
                                                 100%, with the streamer
                                                 fit SOC% = raw*0.385 +
                                                 3.8. Saturates at 250
                                                 in soc-100-idle.asc.
                                                 Slope is provisional —
                                                 only the top 10% of
                                                 the range has been
                                                 observed; a deeper-
                                                 discharge capture is
                                                 needed to pin down the
                                                 LSB and confirm
                                                 linearity below 90%.
      byte 6 (data[5]): 0xFA                  — constant. = 250 raw.
      byte 7 (data[6]): 0x14                  — constant. = 20, the
                                                 SERIES CELL COUNT.
                                                 Cross-check on topology.
      byte 8 (data[7]): 0x00                  — constant.

    Pack current — CONFIRMED:

      pack_current_A = (be16(data[2], data[3]) - 0x7D00) * 0.1

      i.e. the field is signed pack current as a 16-bit big-endian
      value with a fixed bias of 0x7D00 (=32000 raw) and 0.1 A/bit.
      Convention: positive = drawing from pack, negative = charging
      into pack.

      How the earlier "byte 4 is NOT pack current" reading went
      wrong: at idle data[2] sits at 0x7D, so the bias cancels and
      data[3] alone happened to read out as 0..25.5 A — which
      matched the dashboard at rest by coincidence. The moment real
      current crossed ~25.6 A, data[2] ticked to 0x7E and data[3]
      rolled back near zero, making the byte-4-only decode appear
      to drop ("saturate") under load. data[2] = 0x7F is reached in
      driving.asc (peak ≈ 63.6 A under this decode, vs the operator-
      reported dashboard peak of "72" — within sensor / sampling-
      window error).

      Empirical fit:
        * loader-arms.asc idle:        0x7DC4 → +19.6 A   (matches
                                                            dashboard at
                                                            rest)
        * loader-arms.asc raising arm: 0x7E70 → +36.8 A   (matches
                                                            dashboard ~34 A
                                                            during
                                                            lowering)
        * driving.asc peak:            0x7F7C → +63.6 A   (≈ operator-
                                                            reported 72 A)
        * loader-arms.asc / loader2.asc data[2] never drops below 0x7D
          while load was hydraulic-only, so no sign change is needed
          in those captures. driving.asc also stays non-negative
          (regen, if any, is small).

      Dashboard-confirmed calibration set (amp-*.asc, 2026-05-09).
      Each capture is steady-state at a single dashboard-displayed
      amperage; mean decoded current matches the displayed value to
      within ~1 A across the full 0–60 A range. This is the cleanest
      cross-validation of the encoding to date:

        | file          | data[2..3] range  | mean decoded A | dash A |
        |---------------|-------------------|----------------|--------|
        | amp-1.asc     | 0x7D12 (constant) |   1.8          |   1    |
        | amp-18.asc    | 0x7D9D – 0x7DC5   |  17.6          |  18    |
        | amp-35.asc    | 0x7E53 – 0x7E94   |  37.0          |  35    |
        | amp-42.asc    | 0x7E8E – 0x7EC5   |  41.7          |  42    |
        | amp-58.asc    | 0x7F32 – 0x8061   |  62.1          |  58    |

      Notes on this set:
        * amp-1.asc is the only "true idle" snapshot we have on this
          bus — tractor in neutral, key on, no load. data[2..3] is
          *constant* at 0x7D12 across the whole capture (1.8 A
          standby draw for BMS / dashboard / DC-DC). This is much
          lower than the ~17 A seen in earlier "idle in DRIVE"
          captures (drive-fwd, drive-twd, headlights-off-on first
          5 s), confirming that putting the tractor in DRIVE
          energizes inverter / contactor circuitry that pulls
          ~16 A above pure standby.
        * amp-58.asc is the highest-current capture in the set and
          crosses the 0x7F → 0x80 high-byte boundary, exercising
          the second high-byte rollover and confirming the encoding
          extends correctly past 76.7 A.
        * The amp-N "N" in each filename is the operator's read
          off the dashboard at the time of recording, intended as
          rounded ground truth — not a precise calibration marker.
          Sub-amp discrepancies between mean decoded A and dash A
          are expected from rounding / averaging.

      The earlier hypothesis that the dashboard's current came from
      FF21CA byte 2 was wrong — that byte is part of the RPM low
      half (see FF21CA decode section). The operator-confirmed
      reconciliation moment was the loader test on 2026-05-09:
      raising/lowering the loader hydraulics (motor controller
      silent, so FF21CA can't be the source) clearly drives a
      varying current on the bus, and the only signal that tracks
      it is F100F3 bytes 3-4 BE.

    Byte 5 (data[4]): BMS-published SOC — see byte-table entry above.
      The "transient drop to 0xF8" in some loader-arms frames is
      consistent with a SOC tick down during sustained discharge,
      not a separate status bit.

    Byte 6 (data[5]): ~0xFA = 250 throughout. Plausibly a BMS-published
      flag / limit field. TENTATIVE.

    Byte 7 = 0x14 (20): matches the series cell count exactly. Either
      the BMS broadcasts cell count as a sanity field here, or it
      coincidentally tracks. Across all captures it never changes.

F104F3 — "BMS temp min/max summary"
    Pack-wide hottest/coldest module-temperature summary, analogous to
    F102 for cells. Not currently parsed into its own CSV.

F106F3 — "BMS state (charger-dependent)"
    Periodic frame whose contents differ between idle, driving, and
    charging captures. Likely BMS state machine + flags. Across the
    asc/*.asc captures (all 100% SOC, all neutral):
      * driving captures (full-throttle, ft+hyd, accel/decel):
          b0=0x45  b1=0xE0  b2=0xFC  b3..b7=0xFF
      * ignition-with-charger-inserted:
          b0=0x80  b1=0xC4  b2=0xFC  b3..b7=0xFF
      * normal-ignition (during wake-up): b0 transitions through
          0x00, 0x80, 0x45 over the boot sequence.
    Strong "BMS-side running mode" candidate — pairs with the F100D0
    byte-0 mode field on the VC side. Vocabulary so far:
      * 0x00 → init/boot
      * 0x80 → standby / charger-detected
      * 0x45 → ready / driving
    More states (Calibrating, Charging, Discharging, Fault, Sleep)
    expected per the vendor GUI. TENTATIVE.

F107F3 — "BMS current/voltage limits"
    Standard J1939 BMS "limits" frame layout (charge current limit,
    discharge current limit, voltage limits). Now with byte-level
    state evidence from asc/*.asc:
      * driving:          38 A4 27 10 01 76 00 00
      * charger-inserted: 27 10 27 10 00 00 00 00
    Observations:
      * bytes 2..3 = 0x2710 = 10000 in EVERY capture — almost certainly
        a J1939 "not-available" sentinel (10000 in 0.1 A/bit units is
        suspicious; 0xFFFF would be the more conventional sentinel).
      * bytes 0..1 are 0x2710 (= 0 effective limit) when charger is
        inserted but vehicle isn't moving, and 0x38A4 (= 14500) when
        driving. A current-limit field that goes from "no current
        allowed" to "1450 A allowed" would make sense for a high-power
        traction LSB; 0.1 A/bit gives 0 → 1450 A which matches the
        300 A pack rating with margin. TENTATIVE.
      * bytes 4..5 are 0x0176 = 374 driving, 0x0000 charger-inserted.
        374 × 0.2 V/bit = 74.8 V, very close to the rated nominal
        72 V — plausibly a voltage limit / HV cutoff. TENTATIVE.
    Confirming this requires a charge capture from low SOC where the
    BMS will publish meaningful charge-current limits.

F108F3 — "BMS active fault bitmap" (CONFIRMED via injection 2026-05-10)
    Active BMS fault flags. All bytes are 0x00 in healthy idle, verified
    against asc/bms-error-codes/idle-no-bms.asc (key on, neutral, no
    charger). Every per-bit assignment below was established by spoofing
    F108 with the corresponding single bit set and reading the resulting
    code off the dashboard (see solectrac-inject-f108.py and the
    f108-mapping.csv / f108-byte7.csv / f108-byte6.csv result files).

    BYTES 0..6 — mixed encoding (2-bit for bytes 0..3, 1-bit for bytes 4..5)
    ------------------------------------------------------------------------
    Bytes 0..6 don't use a uniform per-bit-rate. The actual layout — from
    a complete 56-bit injection sweep — is:

        byte 0: 2 bits/code -> codes 100, 101, 102, 103
        byte 1: 2 bits/code -> codes 104, 105, 106, 107
        byte 2: 2 bits/code -> codes 108, 109, 110, 111
        byte 3: 2 bits/code -> codes 112, 113 (bits 4..7 SILENT — codes
                               114, 115 are reserved per the manual and
                               take zero bits, not a 2-bit slot)
        byte 4: 1 BIT/code  -> bit 0=116, bit 1=117, bit 2=118, bit 3=119,
                               bit 4=120, bit 5=121, bit 6=122, bit 7=123
        byte 5: 1 bit/code  -> bit 0=124, bit 1=125, bit 2=126, bit 3=127
                               (bits 4..7 silent)
        byte 6: fully silent — all 8 bits produce no dashboard response

    For the 2-bit bytes the dashboard treats either bit of a pair as the
    code being asserted (SAE J1939 "2-bit status" convention: 00=off,
    01/10/11=on at varying severity, dashboard renders any non-00 pair
    as the code on). For the 1-bit bytes each bit independently asserts
    its own code.

    The transition from 2-bit (codes 100..113) to 1-bit (codes 116..127)
    is at the byte 3/4 boundary, with reserved codes 114, 115 occupying
    no bits at all (byte 3 bits 4..7 silent).

    Result files: f108-mapping.csv (byte 0 full, byte 1 partial),
    f108-bytes1to5.csv (bytes 1, 2, byte 3 partial), f108-bytes3to5.csv
    (bytes 3..5 full), f108-byte6.csv (byte 6 silent confirmed),
    f108-byte7.csv (byte 7 — see below).

    The two pre-injection captures decode cleanly under this layout:

      * `bms-fullcharge-102-109-140.asc` (operator-confirmed cycling 102,
        109, 140) F108 = `10 00 04 00 00 00 00 01`:
            byte 0 = 0x10 -> bits 4-5 (pair 2) -> code 102  ✓
            byte 2 = 0x04 -> bits 2-3 (pair 1) -> code 109  ✓
            byte 7 = 0x01 -> code 140 (per byte-7 mapping below)
        Combined: {102, 109, 140} — exactly the operator's set.

      * `bms-124-140-142-143-144-146.asc` (operator-confirmed cycling
        124, 140, 142, 143, 144, 146) F108 = `00 00 00 00 00 01 00 BB`:
            byte 5 = 0x01 -> bit 0 -> code 124  ✓
            byte 7 = 0xBB -> {140, 142, 143, 144, 145}
        Combined: {124, 140, 142, 143, 144, 145}. The operator listed
        ..., 146, but 146 has no source anywhere in F108 — this is
        almost certainly a 145 transcription, not a missing decode.

    BYTE 7 — 1 bit per code, with gaps and a duplicate
    --------------------------------------------------
    Byte 7 is a 1-bit-per-code bitmap covering codes 140 and 142..145.
    The full mapping per f108-byte7.csv:

        bit 0 (0x01) = code 140     System fault level
        bit 1 (0x02) = (silent)     (dashboard shows nothing)
        bit 2 (0x04) = (silent)     (dashboard shows nothing)
        bit 3 (0x08) = code 142     BMS fault need maintenance
        bit 4 (0x10) = code 143     Battery fault need maintenance
        bit 5 (0x20) = code 144     Battery system fault needs maintenance
        bit 6 (0x40) = code 144     duplicate of bit 5 (re-verified)
        bit 7 (0x80) = code 145     Full charge/discharge cycle needed

    Notable: code 146 ("Maintenance mode status") is NOT encoded in byte 7
    or anywhere else in F108 under injection. Bit 6 genuinely re-asserts
    code 144 — the duplicate was re-tested with a single-bit injection and
    the dashboard displayed 144 again, not 145 or 146. Two bits asserting
    the same code might be a warning-vs-fault severity pair the dashboard
    renders identically. Bits 1 and 2 are silent on the dashboard but
    might still carry internal flags that don't surface as numeric codes.

    Code 124 is in byte 5 (not byte 7); the bytes-0..6 and byte-7 active
    sets are merged and deduplicated by the decoder.


PGN decode notes (charger, SA = 0xE5)
-------------------------------------

FF50E5 — "Charger telemetry (V, A)"
    Proprietary B frame from the on-board charger. Byte numbering below
    is 1-based with explicit data[N] (0-based) annotations, matching
    the convention used in the F100F3, F102F3, F108F3, and FF21CA
    sections above.
      * byte 1 (data[0]) = charger status / mode:
          0x00 = idle (not delivering)
          0x01, 0x02 = transient handshake states (only seen briefly
                        during wake-up / ramp; treat as not-delivering)
          0x03 = ACTIVELY delivering charge (CONFIRMED)
      * bytes 2..3 (data[1..2]) LE u16 = output voltage. Same encoding
        as F100F3 byte 2 (data[1]): V = raw * 0.1 + 76.8. CONFIRMED.
      * bytes 4..5 (data[3..4]) LE u16 = output current at 0.1 A/bit
        (no offset). CONFIRMED.

    Status decode CONFIRMED from charging-120V-90ish-to-100.asc — a
    real charging trace from ~90 % SOC with the charger on 120 V mains.
    Status sat at 0x03 for 2863/3007 frames (the active-delivery
    portion); data[1..2] and data[3..4] are only meaningful while
    status == 0x03. The voltage and current scales were anchored
    against contemporaneous F100F3 readings using time-aligned
    regression across those 2863 active-charging frames:
      * data[1..2] vs Pack_V: slope 0.1024, intercept 77.04, R^2 = 0.9856
        → factor 0.1, offset 76.8 V (identical to F100F3 byte 2 / data[1]).
      * data[3..4] vs |Pack_I|: slope 0.0989, intercept -0.07, R^2 = 0.9985
        → 0.1 A/bit, no offset.
    End-to-end the decoded current showed a textbook CC→CV taper
    (~18.5 A → ~9.9 A → ~2.9 A) at constant ~83 V over the capture.

    The earlier "raw 0x00DC ≈ 73.3 V via 1/3 V/bit" hypothesis was
    WRONG — that decoder predates the charging trace and the regression
    against F100. With the corrected formula, raw values seen at idle /
    handshake are not voltage readings at all (see refinement below).

    Refinement from ignition-with-charger-inserted.asc: with the
    charger plug physically inserted in the tractor's port BUT no AC
    mains supplied to the charger, FF50E5 still beacons at ~10 Hz
    (27 frames over 27 s) with status = 0x00, v_raw = 2 and
    i_raw = 2048 (constant). Separately, in normal-ignition.asc
    (no charger connected at all) the charger module ALSO beacons
    during wake-up, with status briefly cycling 0x00 → 0x01 → 0x02
    and v_raw / i_raw sweeping nonsensical values for a few frames.
    These are charger-module self-test artifacts, NOT real bus
    readings — they will inflate the parser summary's V/I range
    unless filtered by status. So:
      * status = 0x00 means "not actively delivering" — does NOT
        distinguish "charger absent" from "charger present but not
        powered" from "charger present, AC live, but BMS-inhibited
        at 100% SOC". Charger-presence detection therefore lives
        elsewhere (current best candidate: F108F3 byte 7).
      * status 0x01/0x02 are transient handshake states. Treat as
        not-delivering for power accounting.
      * Don't trust V/I unless status == 0x03.

    RPM — CONFIRMED:

      rpm = ((data[3] << 8) | data[2]) - 0x0C80
      Equivalently: rpm = (data[3] - 12) * 256 + (data[2] - 128).

      Empirical fit:
        * Idle (motor commanded zero):
            data[2] = 0x80, data[3] = 0x0C → 0 RPM.
        * Full throttle (pedal pegged, in neutral):
            data[2] sweeps 0x30..0x40 with data[3] = 0x16 →
            ~2480..2496 RPM. Operator reports the dashboard's RPM
            gauge sat near 2500 at this moment, matching to within
            quantization.
        * Accelerate-decelerate (varying throttle, in neutral):
            time-resolved trace shows a textbook ramp from 0 RPM,
            through the full 0..2500 range over ~13 s of acceleration,
            steady cruise at ~2488 RPM, then a symmetric ramp back
            down to 0 RPM. data[3] (the high byte) advances cleanly
            at every multiple-of-256 RPM boundary, exactly as a 16-bit
            value should.

      The 0x0C80 bias is the controller's "zero RPM" sentinel.
      In practice the field appears to be **magnitude only** — see
      "Direction encoding" below — so values below 0x0C80 (which would
      represent negative / reverse rotation under an offset-binary
      reading) are not actually emitted, even when the operator
      drives in reverse. Reverse direction is signaled instead by
      data[7]; the 0x0C80 offset is best understood as a fixed
      configuration constant rather than a true signed-value zero.

    Direction + range encoding — CONFIRMED:

      data[7] is a packed transmission-state byte:

          high nibble (data[7] >> 4)  = range gear
              0x0 = Range 1
              0x1 = Range 2
              0x2 = Range 3

          low nibble  (data[7] & 0xF) = F/N/R lever
              0x0 = Neutral
              0x4 = Forward
              0x8 = Reverse

    Reconciliation with earlier "data[2] = signed pack current"
    interpretation:

      An earlier reading of these notes interpreted data[2] alone as
      a signed pack-current field with 128 = 0, 1 A/bit, citing the
      cross-check that data[2] = 57 → signed -71 ≈ a dashboard
      reading of "72". That interpretation does not survive the
      operator-confirmed RPM = 2500 at full throttle:

        * At full throttle in neutral data[2] sweeps 0x30..0x40 (48..64).
          Under "data[2] = signed current" that's -64..-80 A. An
          unloaded motor at ~2500 RPM in neutral does not draw 70+
          amps; the running friction of an electric drive is on the
          order of single-digit amps. The signal magnitude doesn't
          fit a current reading under this load condition.
        * data[3] varies systematically with throttle (0x0C at idle
          → 0x16 at full throttle). Under "data[3] = state /
          sub-counter" the smooth ramp has no clean explanation.
          Under "RPM high byte" it's exactly the carry from
          data[2] crossing 256-boundaries.
        * data[2..3] LE - 3200 reproduces 0 RPM at idle and ≈2488
          at full throttle to within quantization. Both the
          single-byte current reading AND the RPM reading would
          make data[2] sweep the full range during accel-decel,
          but only the RPM reading explains data[3] simultaneously.

      "72") was a coincidence: FF21CA data[2] is just the RPM low

      resolves this — hydraulic load with the motor controller
      silent showed a clean varying current on F100F3 only, and
      the byte-3-only decode misbehaved exactly as predicted by
      the 16-bit BE biased encoding.

Motor-controller error codes — SOLVED 2026-05-13
------------------------------------------------

**Resolution: MC error codes ride standard J1939 DM1 (PGN FECA from SA
0xCA). The SPN field value equals the dashboard-displayed MC code; the
cluster prepends "MC" based on source address.**

Confirmed via live CAN injection (`util/mc_inject.py`) at slcan @ 250 kbaud,
ID 0x18FECACA, with single-DTC payloads:

  SPN 12 (0x0C) -> dashboard "MC12" (Controller Over Current)
  SPN 47 (0x2F) -> dashboard "MC47" (HPD/Sequencing Fault)
  SPN 99 (0x63) -> dashboard "MC99" (Parameter Mismatch)

A populated DM1 from SA 0xF3 (BMS) was ignored by the cluster, proving
the cluster has subsystem-specific decoders rather than a unified DM1
path:
  - MC (SA 0xCA): J1939 DM1, SPN = displayed number.
  - BMS (SA 0xF3): proprietary F108 bitmap (continuous broadcast).

**Latch quirk:** the cluster latches DM1 DTCs on receipt and does NOT
unlatch when DM1 returns to empty. Standard J1939 says DTCs go
"previously active" after 3 s of frame absence; this cluster keeps them
on screen until a key cycle. When iterating on injection tests, key-
cycle between probes so you can tell whether a new code came from the
new injection or from a stale latch.

**Why empty in every capture:** every prior asc/*.asc capture was taken
under nominal operation — no MC faults ever occurred. DM1 was the right
channel all along; we just never saw it populated. (Same situation that
hid F108 byte 7 before the operator deliberately produced BMS faults.)

**FF21CA bytes 2 and 7 are NOT fault fields.** Injecting non-zero
values into FF21CA byte 7 flashed dashboard lamps but never produced a
numeric code. Byte 7 is read by the cluster, likely for lamp / status
flags (analogous to DM1 byte 0's MIL/RSL/AWL/PSL layout), but does not
carry the code number. Bit-layout for FF21CA byte 7 (and possibly byte
2) is a worthwhile future probe.

Code numbering ranges remain non-overlapping: MC = 12..99 (per manual),
BMS = 100..146 (per manual). A dashboard "code 47" is unambiguously MC;
"code 124" is unambiguously BMS.

Historical methodology notes (kept for context):
------------------------------------------------------------

What's already on the bus from SA 0xCA:
* FECA (DM1)  — empty (`00 00 00 00 00 00 FF FF`).
* FF21CA      — motor telemetry (~85 Hz). Three bytes are still
                undecoded and are the highest-value MC-fault candidates:
                  * byte 2 (data[1]): 0x00 constant in all captures
                  * byte 5 (data[4]): three-state 0x28 / 0x3B / 0x3C
                                      (changes near startup calibration)
                  * byte 7 (data[6]): 0x00 constant in all captures
                A constant-zero byte that "lights up" only under fault
                conditions is exactly the F108-byte-7 pattern. byte 7 is
                the strongest single candidate; byte 2 next; byte 5 is
                more likely a state-machine field than a fault bitmap
                (it changes during normal startup, which a fault flag
                shouldn't).
* Possibly a separate proprietary PGN we haven't seen because it's
  silent until a fault asserts. Worth enumerating *all* (PGN, SA=0xCA)
  pairs across every existing asc/*.asc to confirm FECA + FF21CA is
  really all 0xCA emits in the no-fault state.
* A capture might also surface MC fault info via a non-0xCA proxy
  (the dashboard or VC re-broadcasting); enumerate any *new* PGNs
  that appear in a fault capture but not in any clean baseline.

Capture procedure (mirrors the BMS pair that nailed down F108):

1. Clean baseline in the same key state as the planned fault capture.
   Key on, in DRIVE (not neutral — the MC has more state machinery
   awake in DRIVE; FF21CA byte 5 shifts), no faults active. Call it
   `mc-clean-drive.asc`.
2. Fault capture with one (ideally just one) MC code on the dashboard.
   Single-code is critical: with the BMS we got lucky (six set bits in
   numeric order) and could deduce the layout from one popcount match.
   With a single set bit, the encoding is unambiguous regardless of
   layout. Operator menu / cycle behavior of the dashboard determines
   how to isolate one code.
3. Byte-level diff between the two captures, restricted to messages
   from SA 0xCA (and any new PGN that appears only in the fault file).
   The change set should be small.
4. Hypothesis check against the operator manual's MC error-code table
   (assuming Solectrac publishes one analogous to the BMS table).
   Popcount of the set bits should equal the count of displayed codes;
   the bit-to-code mapping should fall out of single-code captures.

Things to watch for:

* FF21CA goes silent during charging (contactors open for traction).
  Captures must be in a traction-ready state — fault captures while
  plugged in won't carry FF21CA at all.
* MC error code numbering may not share a numeric range with the BMS
  table. A code 124 on the dashboard could in principle be either
  BMS:124 or MC:124 depending on which subsystem the dashboard is
  annunciating. Confirm against context (which subsystem the manual
  attributes the code to) before reading too much into the number.
* If the diff comes back empty (no FF21CA bytes change, no new PGNs
  from 0xCA appear, no proxy frames change), that's the answer —
  MC error codes aren't on this segment. They're then either:
    - dashboard-synthesized from telemetry (the dashboard sees the
      same FF21CA / F1xx telemetry we do and could compute codes
      against local thresholds without the BMS or MC broadcasting
      them), or
    - on the un-tapped drivetrain bus.
* Don't intentionally induce dangerous faults to get a capture.
  Reasonable MC faults to wait for or trigger safely: motor
  over-temperature from sustained hydraulic-blocked operation,
  low-pack-V cutoff during depleted-pack driving, direction-switch
  contention from rapidly toggling forward/reverse pedals. Avoid
  anything involving disconnecting sensor harnesses or stalling
  the motor against a hard mechanical stop.

Triage order if a fault capture is obtained:

1. Diff FF21CA byte 7 first. Constant-zero across all clean captures
   → highest prior probability of being the fault bitmap.
2. Diff FF21CA byte 2 next. Same reasoning, weaker prior because
   nothing else suggests it carries semantics.
3. Enumerate any new (PGN, SA=0xCA) pairs only present in the fault
   capture.
4. Enumerate any byte in any frame whose value range expanded versus
   the clean baseline.
5. Only after the above turn up nothing should the un-tapped-second-
   bus or dashboard-synthesized hypotheses be advanced; both imply
   passive MC-fault capture on this segment is impossible and the
   work shifts to active probing or a second tap.

Decoder errata exposed by the screenshot:

* The **"Cells in parallel: 20"** field on the GUI almost certainly

Confidence summary
------------------

Empirical observations
^^^^^^^^^^^^^^^^^^^^^^

* Resting / standby pack draw (continuous.csv): on the order of
  100–200 W. Consistent with BMS + dash + accessory loads, no traction.
* Driving (driving.csv): peak draw on the order of ~1 kW; typical
  driving current well under nameplate.
* Driving + hydraulics (dph.csv): hydraulic actuation adds clearly
  visible current spikes on top of the drive baseline; peak combined
  draw on the order of ~1.8 kW. The mean of dph.csv is *lower* than
  driving.csv only because dph.csv includes more idle/parked time —
  per-event peaks are higher in dph.
* 100% SOC reference (asc/*.asc): all 5 captures show cell means
  in the 4153–4165 mV band (8–12 mV cell-to-cell within a single
  capture; 6–8 mV F102 spread). Pack-V estimate (20 × cell mean):
    - idle / no charger:       83.29 V
    - charger inserted, no AC: 83.31 V
    - full throttle (neutral): 83.22 V
    - accel/decel (neutral):   83.18 V
    - full thr + hyd (neutral):83.06 V
  The 0.25 V sag from idle to highest load is the cleanest pack-V
  load response we have on this dataset.
* Charging (charging.csv): charger output on the order of ~1.5 kW into
  the pack at this SOC. Pack current reported by F100F3 bytes 3-4 BE
  matches charger current from FF50E5 within rounding (sign convention:
  charging shows up as the raw 16-bit value sitting below 0x7D00),
  which is what cross-validates the 0.1 A/bit scaling.
* Cell spread at rest: tens of mV. No outlier cells observed.
* No active DM1 trouble codes during any capture (FECA frames are
  emitted but their SPN/FMI fields are inactive).
* Temperatures: all populated channels read 12–15 °C, consistent with
  the ~13 °C ambient. No self-heating signature in any capture (none
  of these runs are long or hard enough to warm the pack).

Decoder confidence
^^^^^^^^^^^^^^^^^^

CONFIRMED:

    module-temp scaling (u8 + 40 °C)
    pack topology: 20S, 7 temp probes (vendor GUI)
